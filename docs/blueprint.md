# Next.js Template Blueprint

> **Agent action required:** Read `docs/agents.md` now and create `submission.md` in the project root if it does not exist.

This project is a CreatorEngine runtime wrapped in a Next.js app-router application. Use this document as the context anchor before making code changes, especially when working with an agentic AI assistant.

## 1. High-Level Mental Model

The project has two halves:

- Editor-authored content: `project.bjseditor`, `assets/`, and scripts attached to scene objects.
- Next.js runtime: `src/app/**`, Babylon.js engine setup, `loadScene`, `scriptsMap`, and generated `public/scene`.

Scene composition belongs in the editor side. App shell, routing, overlays, deployment, and web UI belong in the Next.js side. Gameplay or object behavior should usually live in `src/scripts/**` so it can stay attached to editor objects.

## 3. File Responsibilities

1. `project.bjseditor`
   - Editor manifest.
   - Keeps the project associated with CreatorEngine.
   - Stores package manager, plugin, texture compression, and last-opened scene metadata.

2. `assets/`
   - Source-of-truth authoring folder.
   - Contains editor scenes, material files, textures, environment files, and source assets.
   - Prefer editor-driven changes for scene composition and object attachment.

3. `public/scene/`
   - Generated runtime folder.
   - Created by CreatorEngine or `yarn generate`.
   - Should be treated as build output, not hand-authored source.

4. `src/app/page.tsx`
   - Main client-side Babylon.js runtime.
   - Owns canvas, engine, scene, physics setup, scene loading, camera controls, render loop, and cleanup.

5. `src/app/layout.tsx`
   - Next.js root layout.
   - Good place for page metadata, fonts, global wrappers, and providers.
   - Not the right place for Babylon.js rendering because it can be server-rendered.

6. `src/app/globals.css`
   - Global styles and Tailwind setup.
   - Keep canvas layout rules predictable so the engine resize behavior remains stable.

7. `src/scripts.ts`
   - Generated script registry.
   - Bridges editor-attached script paths to imported TypeScript modules.
   - Do not manually maintain this for normal feature work.

8. `src/scripts/**`
   - Runtime behavior scripts attached to scene objects through the editor.
   - Best home for gameplay, interaction, object animation, stateful scene logic, and inspector-exposed values.

9. `scripts/compress-for-s3.js`
   - Deployment helper for compressed build artifacts.
   - Keep separate from runtime scene behavior.

10. `next.config.js`, `tailwind.config.ts`, `postcss.config.js`, `tsconfig.json`
   - Framework and tooling configuration.
   - Change deliberately because path aliases, bundling, and static asset behavior can affect scene loading.

## 4. Runtime Entry Point

The Babylon.js app starts in `src/app/page.tsx`.

Key responsibilities:

- Mark the page as a client component with `"use client"`.
- Create the `Engine` from the canvas ref.
- Create the `Scene`.
- Enable Havok physics.
- Load the generated scene with `loadScene("./scene/", "example.babylon", scene, scriptsMap, { quality: "high" })`.
- Attach the active camera controls.
- Start `engine.runRenderLoop`.
- Dispose the scene, engine, and resize listener when React unmounts.

Do not move Babylon.js engine creation into a server component. Anything touching `window`, `document`, canvas, or Babylon.js rendering must remain client-side.

## 5. Editor Contract

`project.bjseditor` lets CreatorEngine recognize this folder as an editor project. It points at the current authoring scene in `assets/example.scene`.

`assets/` is the source of truth for editor-authored content.

`public/scene/` is generated runtime output. Do not hand-edit it for lasting changes.

`src/scripts.ts` is generated and maps editor-attached scripts to imports. Add or edit behavior in `src/scripts/**`, then regenerate.

## 6. Runtime Loading Contract

The runtime loading contract is:

1. Create a canvas-backed Babylon.js `Engine`.
2. Create a Babylon.js `Scene`.
3. Enable physics and scene-level systems before loading the editor scene.
4. Keep required Babylon.js side-effect imports so serialized editor content can deserialize correctly.
5. Call `loadScene("./scene/", "example.babylon", scene, scriptsMap, { quality: "high" })`.
6. Attach active camera controls after the scene is loaded.
7. Start one render loop for the engine.
8. Dispose the scene, engine, and browser event listeners during React cleanup.

If an agent changes any of these steps, it should explain why and test both development and production builds.

## 7. Scripts And Scene Behavior

Use `src/scripts/**` for behavior attached to scene objects.

The default example script is `src/scripts/box.ts`. It implements `IScript` from `babylonjs-editor-tools` and rotates a mesh during `onUpdate`.

If a new script is added by code, it still needs to be attached to an object in CreatorEngine or included in the generated script map before it will run.

Script lifecycle guidelines:

1. Use constructors to receive the Babylon.js object passed by the editor.
2. Use `onStart` for initialization that depends on the loaded scene.
3. Use `onUpdate` for per-frame logic.
4. Use Babylon.js animation ratio or delta time for frame-rate-independent motion.
5. Use `visibleAsNumber` and related editor decorators for values users should tune in the inspector.
6. Keep script files small and object-focused.
7. Avoid putting broad app routing or React UI state inside scene scripts.

## 8. XR, VR, MR, AR, And Immersive Features

If the user asks for XR, VR, MR, AR, mixed reality, virtual reality, headset support, passthrough, hand tracking, spatial input, immersive mode, or Meta Quest support, use the Meta/Facebook Immersive Web SDK dependency declared as `@iwsdk/core`.

Agent rules for immersive requests:

1. Treat Babylon.js as the scene, rendering, asset, physics, and editor-content layer.
2. Treat `@iwsdk/core` as the preferred immersive platform integration layer.
3. Do not invent a separate XR framework unless the user explicitly asks for one.
4. Keep the existing `loadScene` and `scriptsMap` flow intact.
5. Keep `@babylonjs/core/XR/features/WebXRDepthSensing` unless a verified SDK integration replaces the need for it.
6. Add immersive setup only in client-side code because Next.js server components cannot access browser XR APIs.
7. Put reusable immersive behavior in `src/scripts/**` when it belongs to scene objects.
8. Put app-level session buttons, permission prompts, and overlays in React client UI.
9. If exact `@iwsdk/core` APIs are unknown in the local checkout, inspect installed package docs or types after dependencies are installed before coding.
10. Test fallback behavior for browsers or devices without XR support.

Recommended immersive implementation path:

1. Confirm `@iwsdk/core` is installed from `package.json`.
2. Inspect its local README, type declarations, or examples.
3. Add a small client-side adapter module instead of scattering SDK calls through `page.tsx`.
4. Initialize immersive support after the Babylon.js scene and camera exist.
5. Keep normal desktop canvas rendering working when immersive mode is unavailable.

## 9. Commands

Run these from this template folder:

```bash
yarn dev
yarn generate
yarn build
yarn start
```

- `yarn dev`: starts Next.js development server.
- `yarn generate`: runs `babylonjs-editor-cli pack` and creates or updates `public/scene`.
- `yarn build`: builds the Next.js app.
- `yarn build:compress`: builds and runs `scripts/compress-for-s3.js`.
- `yarn start`: starts the production Next.js server after a build.

Always run `yarn generate` before `yarn build` when scene assets or editor scripts changed.

## 11. Common Changes

Add gameplay:

1. Add code in `src/scripts/**`.
2. Attach the script in CreatorEngine.
3. Run `yarn generate`.
4. Test with `yarn dev`.

Add UI:

1. Keep the canvas mounted in `src/app/page.tsx` or a client child component.
2. Add React UI around or over the canvas.
3. Keep resize handling so `engine.resize()` runs.

Change deployment path:

1. Check `next.config.js`.
2. Verify `loadScene("./scene/", ...)` still resolves to generated assets.
3. Build and test the static asset URL.

Add immersive mode:

1. Keep `src/app/page.tsx` as a client component.
2. Inspect `@iwsdk/core` local types or docs after install.
3. Create a client-side immersive adapter.
4. Connect it to the existing Babylon.js scene and camera.
5. Preserve non-immersive desktop rendering.

### 12.2 Core Interaction Loop

Define the repeated loop before implementing features.

Use this format:

1. User does an action.
2. Input layer receives it.
3. A system interprets it.
4. One or more entities change.
5. The scene renders feedback.
6. UI communicates state if needed.
7. The user decides the next action.

Example:

1. User presses movement keys.
2. Input system records direction.
3. Movement system computes velocity.
4. Player entity moves.
5. Camera follows or scene updates.
6. UI updates stamina or speed if present.

Agent instruction: do not implement a feature until its loop is clear enough to identify input, system, entity, scene feedback, and UI feedback.

### 12.3 Constraints

Document constraints early.

1. Performance target:
   - Default target is 60 FPS on a modern desktop browser.
   - For headset or immersive work, prefer stable frame time over visual complexity.
   - Avoid expensive per-frame allocations.

2. Scope:
   - Keep hackathon or prototype features small.
   - Build thin vertical slices before broad systems.
   - Prefer one working interaction over many partial interactions.

3. Runtime:
   - This template runs in Next.js client components.
   - Babylon.js rendering must stay browser-side.
   - Do not put rendering logic in server components.

4. Offline or backend:
   - Do not add a backend unless requested.
   - Prefer local state for prototypes.
   - Keep save or sync features isolated behind service modules.

5. Asset constraints:
   - Editor-authored scene assets belong in `assets/`.
   - Generated runtime assets belong in `public/scene/`.
   - Do not hand-edit generated scene output for permanent changes.

### 12.4 Non-Goals

Write non-goals to prevent feature creep.

Common non-goals:

1. Do not rebuild the CreatorEngine inside the runtime app.
2. Do not replace editor scene loading unless explicitly requested.
3. Do not create a new engine abstraction before there are multiple real use cases.
4. Do not add multiplayer, persistence, auth, payments, or backend systems unless requested.
5. Do not mix all feature logic into `src/app/page.tsx`.
6. Do not add XR, VR, MR, or AR with a random library when `@iwsdk/core` is declared for immersive work.
7. Do not optimize before the feature is understandable and measurable.

Agent instruction: when a request implies a non-goal, call it out and choose the smaller implementation unless the user confirms the larger scope.

2. Entity layer:

3. Systems layer:
   - Movement.

   - State transitions.
   - XR session coordination.

4. UI layer:
   - React components.

5. Integration layer:

Rule: keep responsibilities separate. A scene should not become a giant gameplay manager. A UI component should not directly mutate many meshes. A script should not secretly own global application routing.

### 12.6 Data Flow

Use this default flow:

1. User input occurs.
2. UI or input module captures it.
3. System reads input and current state.
4. System updates entities.
5. Entities affect scene transforms, materials, animations, or metadata.
6. Scene renders.
7. UI reflects important state.

For editor-attached scripts:

1. Editor attaches script to object.
2. `yarn generate` updates `src/scripts.ts`.
3. Runtime loads scene with `loadScene`.
4. `babylonjs-editor-tools` applies scripts from `scriptsMap`.
5. Script lifecycle methods run.

Agent instruction: preserve this flow unless the user asks for a different architecture.

### 12.11 System Design Pattern

Every reusable feature should be describable as a system.

System template:

```text
Name:
Purpose:
Layer:
Inputs:
Outputs:
Entities affected:
Dependencies:
Update frequency:
Failure behavior:
Test plan:
```

Example:

```text
Name: PlayerMovementSystem
Purpose: Move a player mesh from keyboard input.
Layer: System plus entity script.
Inputs: WASD keys, delta time, speed.
Outputs: Updated player transform.
Entities affected: Player mesh.
Dependencies: Babylon.js Mesh, Vector3, editor script lifecycle.
Update frequency: Every frame.
Failure behavior: Do nothing if mesh is missing.
Test plan: Press WASD and confirm smooth movement.
```

Agent instruction: for multi-file features, write the system template first in the implementation notes or comments.

### 12.12 Entity Design Pattern

Entities are objects in the world.

Guidelines:

1. Keep entities reusable.
2. Keep identity and state close to the object.
3. Avoid embedding broad game rules directly in entities.
4. Attach behavior through editor scripts or systems.
5. Use metadata only when it improves clarity.
6. Prefer composition over deep inheritance.
7. Name entities clearly in the editor.
8. Keep units and scale consistent.

Entity checklist:

1. What is this object?
2. Who owns it?
3. What scripts are attached?
4. What systems read or modify it?
5. What happens when it is disposed?

### 12.13 Scene Responsibilities

The scene runtime should only:

1. Initialize engine.
2. Initialize scene.
3. Configure physics.
4. Load editor scene.
5. Attach camera controls.
6. Start render loop.
7. Register high-level services only when needed.
8. Clean up resources.

The scene runtime should not:

1. Contain all gameplay logic.
2. Own every UI state transition.
3. Hard-code many object names for unrelated systems.
4. Replace editor-authored content casually.
5. Create multiple render loops.

### 12.14 UI Responsibilities

3. Hide scene lifecycle side effects in unrelated components.
4. Run browser-only code from server components.

### 12.15 Immersive Feature Enforcement

7. Use instances or thin instances for repeated meshes where appropriate.

- Missing materials, lights, shadows, or textures: check Babylon.js side-effect imports in `src/app/page.tsx`.
- XR request got implemented with unrelated APIs: revise to use `@iwsdk/core` as the integration layer.
